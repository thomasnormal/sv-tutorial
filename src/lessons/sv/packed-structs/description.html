Packed structs bundle related signals — write-enable, address, and data — into a single named type. Instead of three separate ports you pass one mem_cmd_t; the receiver unpacks by field name.

Syntax

typedef struct packed {
    logic        <field>;
    logic [N:0]  <field>;
} <name>_t;

Fields are laid out MSB-first in declaration order. You read and write them by name:

mem_cmd_t cmd;
cmd.we   = 1'b1;
cmd.addr = 4'd3;

Or with a struct literal '{...}:

cmd = '{we: 1'b1, addr: 4'd3, wdata: 8'hA5};

'{we: ..., addr: ..., wdata: ...} is a struct literal. Do not confuse it with {a, b, c} which is bit concatenation — both produce the same 13-bit pattern here, but the literal form is much more readable.

Bit layout of mem_cmd_t

The entire command fits in 13 bits. You can also assign or compare it as a raw pattern:

// we=1, addr=5 (4'b0101), wdata=0x4D — same as struct literal
cmd = 13'b1_0101_01001101;

Packages

The type is declared in a package so both the DUT (sram_cmd.sv) and the testbench share exactly one definition. The DUT receives a single mem_cmd_t cmd port and accesses cmd.we, cmd.addr, cmd.wdata by name.

This is the command bus for our memory chip. In Part 3 the UVM driver will assemble a mem_cmd_t from a sequence item and drive it onto the interface every clock cycle.

Exercise

Open mem_cmd.sv. Inside the typedef struct packed { } mem_cmd_t; skeleton, add the three fields of the SRAM command bus (refer to the bit-layout diagram above):

  • A single-bit write-enable flag (we) — MSB, bit 12
  • A 4-bit address (addr) for a 16-entry SRAM — bits 11..8
  • An 8-bit write-data bus (wdata) — bits 7..0

Run the simulation — you should see two PASS lines: one read-back via struct literal, one via raw bit assignment.
