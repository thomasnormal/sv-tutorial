Writing directed test vectors by hand doesn't scale. Constrained random verification lets the solver generate legal stimulus automatically.

class <name>;
  rand <type> <field>;       // solver picks a new value each randomize()
  constraint <name> { <expr>; }  // boolean expression the solver must satisfy
endclass

Call randomize() to generate new values. The return value indicates success — always check it:

assert (t.randomize()) else $fatal(1, "randomize failed");

Constraint blocks

Constraints are declarative — you describe what is legal, not how to pick it:

constraint low_bank_c  { addr inside {[0:7]};   }  // low 8 addresses
constraint write_only  { we == 1;                }  // writes only
constraint no_overlap  { !(addr == 4'd0 && we);  }  // never write address 0

Inline override with with

Add a temporary extra constraint at the call site without modifying the class:

void'(t.randomize() with { addr inside {[8:15]}; });  // high bank for this call only

Disabling constraints

Toggle a named constraint at runtime:

t.low_bank_c.constraint_mode(0);  // disable — full range
t.low_bank_c.constraint_mode(1);  // re-enable

Open rand_txn.sv and complete three tasks:

  1. Declare rand logic [3:0] addr and rand bit we.
  2. Add constraint low_bank_c { addr inside {[0:7]}; }.
  3. In Scenario 2, call t.randomize() with { addr inside {[8:15]}; } and assert addr >= 8.

UVM's uvm_sequence_item uses the same mechanism. The sequence item you'll build in the UVM chapter is exactly this class — with rand bit we, rand logic [3:0] addr, and a read_c constraint — wired up to a driver that calls randomize() before each transaction.
